Structured Output

Generating Reliable Data Formats

Code Framework 10 min read Intermediate

Beyond Free-Form Text

Structured Output prompting focuses on getting AI models to produce data in specific, parseable formats like JSON, XML, or YAML. This is essential for building applications where AI responses feed into downstream systems that expect structured data.

Core Challenge

Language models are trained to produce natural text, not data structures. Structured output requires constraining the model's natural verbosity while ensuring format compliance‚Äîwithout breaking the semantic content.

Common Output Formats

Different formats suit different use cases. Understanding their characteristics helps you choose appropriately.

JSON Output

The most common format for structured AI output. Well-supported by all programming languages.

Prompting Strategies

Techniques to reliably get properly formatted output from language models.

Provide an explicit schema or template showing the expected structure.

Prompt: "Extract contact information from this text. Return as JSON matching this schema:

{
  "name": "string",
  "email": "string or null",
  "phone": "string or null",
  "company": "string or null"
}

Use null for missing fields. Return only the JSON, no other text."

Show a complete example of correctly formatted output.

Prompt: "Categorize each product. Format your response exactly like this example:

[
  {"product": "iPhone 15", "category": "Electronics", "subcategory": "Smartphones"},
  {"product": "Nike Air Max", "category": "Apparel", "subcategory": "Footwear"}
]

Now categorize: MacBook Pro, Levi's 501 Jeans, Sony WH-1000XM5"

Use explicit instructions to prevent natural language additions.

Effective phrases:

  • "Return only valid JSON, no explanation"
  • "Your entire response must be parseable JSON"
  • "Do not include any text before or after the JSON"
  • "Start your response with { and end with }"

Start the assistant's response to constrain the format from the first character.

Prompt: "Extract the data as JSON."

Assistant prefix: "{"

By starting the response with {, you force the model to continue valid JSON rather than adding preamble like "Here's the JSON:"

Common Issues and Fixes

Preamble Text

Issue: Model adds "Here's the JSON:" before the data.

Fix: Use prefix prompting or explicit "Return only JSON" instruction.

Trailing Commas

Issue: Model adds comma after last array/object element.

Fix: Post-process to remove, or use lenient JSON parser.

Unescaped Strings

Issue: Quotes or newlines in string values break JSON.

Fix: Ask model to escape special characters, or post-process.

Schema Drift

Issue: Model adds extra fields or uses different names.

Fix: Provide exact schema and state "use these exact field names."

Retry on Failure

If validation fails, include the error in a retry prompt asking for corrected output.

Retry Prompt Pattern

"Your previous response had this JSON error: [error message]. Please fix the formatting and return valid JSON. Previous attempt: [previous response]"

Best Practices

  • Prefer JSON for most cases - It has the best model support and tooling
  • Always provide a schema or example - Don't rely on the model to infer structure
  • Use explicit field names - Avoid ambiguity with clear, descriptive names
  • Handle missing data explicitly - Specify whether to use null, empty string, or omit
  • Test edge cases - Special characters, empty inputs, very long values
  • Build robust parsing - Expect some failures and handle gracefully
  • Consider API features - Many APIs offer native JSON mode or function calling
